# 烤肉 AI 店长 — 项目路线图

## 现状总结（已完成）

**MVP 前端原型**：Vite + React + TypeScript + Tailwind 的移动端网页原型，模拟一个面向成都连锁烤肉品牌的 AI 运营助手。

- 7 个页面全部实现：今天 / 上新助手 / 门店体检 / 套餐设计 / 内容生成 / 评价管理 / AI 助手
- 所有数据都是 mock：`src/mock/` 存静态种子数据（门店、评价等），`src/ai/` 存"生成器"函数（上新方案、套餐、文案、配图、聊天回复），用模板拼接实现，不调用任何真实 API
- 无后端、无数据库、无鉴权——状态只存在浏览器 `localStorage`
- 已部署：GitHub 仓库 `leoriohope/roast-ai-manager`，GitHub Actions 做 CI（typecheck + build），Vercel 托管，`dev`/`uat`/`main` 三分支对应三个环境

**这一步的定位**：验证产品假设、给试点客户演示用的原型，还不是能商用的产品。

## 后续子项目拆分

按建议顺序排列，每个都可以独立启动、验收；带 → 的是明显的依赖关系。

### 1. 后端服务 + 数据库
把 `localStorage` 换成真正的持久化。搭一个最小后端（推荐 Vercel Serverless Functions，不用额外起服务器）+ 一个数据库（Supabase/Postgres 都行），先把现有的数据结构（门店、任务、生成记录、评价）原样搬过去，先不接真实 AI。
**验收标准**：多个设备/浏览器打开看到同一份数据；刷新、清缓存不丢数据。
**为什么排第一**：后面接真实 AI API 时，API key 不能放在前端，必须经过后端代理——先把后端搭好，第 2、3 步才有地方落地。

### 2. 真实文案生成接入 → 依赖 1
把 `generateLaunchPlan` / `generatePackagePlan` / `generateContentPlan` / `generateAssistantReply` 里的模板拼接逻辑，换成真实调用大模型（比如 Claude API）。
**验收标准**：同样的输入，每次生成的文案内容真的不一样、有实际"理解"输入信息的质量提升，而不是关键词替换。
**改动范围**：主要在 `src/ai/*.ts` 内部实现，之前设计的"seam"应该能让这一步改动面很小。

### 3. 真实图片生成接入 → 依赖 1
把 `generateContentImage` 的 SVG 占位图换成真实图像生成 API（比如 DALL·E、Stable Diffusion 等）。
**验收标准**：生成的图能实际用于宣传（不只是渐变色+emoji）。
**备注**：这个优先级可以比第 2 步低——先把文案做真实，图片可以晚一点，看试点客户反馈里"配图"到底有没有人在意。

### 4. 用户登录与角色区分 → 依赖 1
最初的产品设想里有三种角色（老板/店长/员工），目前原型没有登录，人人看到的都一样。加最基础的登录 + 角色权限：老板看营收和门店对比，店长看今日任务和内容生成，员工可能只需要看话术。
**验收标准**：不同角色登录后默认页面/可见内容不同。

### 5. 真实评价数据接入 → 依赖 1
现在的评价是写死的 mock 数据。这一步是从美团/点评/抖音等平台把真实评价拉进来——大概率没有官方开放 API，可能得先做成"人工导入 CSV/Excel"这种半自动方案，而不是自动抓取（抓取涉及平台合规问题，需要单独评估）。
**验收标准**：评价分析页面上看到的是这个客户门店的真实评价，而不是示例数据。

### 6. 多门店/多品牌支持（多租户）→ 依赖 1
现在 4 个门店是硬编码在 `mock/stores.ts` 里，且整个产品是为"一个 BBQ 品牌"定制的措辞（"帮我做上新"这类文案跟烤肉强相关）。如果要卖给其他类型餐厅或者多个品牌，需要把品牌/门店变成可配置的租户数据，而不是写死。
**验收标准**：新增一个不同品类的门店（比如火锅店），不用改代码就能用。
**备注**：这一步如果只服务这一个试点客户，可以往后放，甚至可能永远不做。

### 7. 访问权限与安全加固
现在部署的几个 URL（prod/dev/uat）是公网可访问、无密码的。等接了真实客户数据（尤其是第 5 步之后），需要：Vercel Deployment Protection（加密码或登录门槛）、后端 API key 通过环境变量管理、必要的速率限制。
**验收标准**：非授权用户打不开 dev/uat，prod 至少有基础访问控制。

### 8.（远期）多 Agent 架构重构
最初产品构想里提到未来可能拆成多个专职 Agent（门店诊断 Agent、上新 Agent、套餐设计 Agent、评价分析 Agent、风险审核 Agent、周报 Agent 等），现在的 `ai/` 目录是扁平的函数集合。等第 2 步真实 AI 接入、且发现单一 prompt 处理不了复杂度时，再考虑要不要往这个方向重构，现在不用提前设计。

---

## 建议的启动顺序

对试点客户演示来说，最有说服力的路径大概是：**1 → 2 → 5**（后端 + 真文案 + 真评价），能让客户看到"这真的是在用我们门店的数据生成有用的内容"，而不是"这是个好看的 demo"。3、4、6、7、8 都可以往后放，按实际反馈决定要不要做、什么时候做。

每个子项目开始前，我们可以单独过一遍这个子项目的具体技术方案（类似之前给 MVP 原型走的 plan mode 流程），确认范围和验收标准后再动手，避免一次性铺太大摊子。
