# 烤肉 AI 店长 — 项目路线图

## 现状总结（已完成，2026-07-12 更新）

**从 mock 原型演进为真实试点系统**：Vite + React + TypeScript + Tailwind 的移动端网页应用，面向成都连锁烤肉品牌（试点客户：南北征赞·野核桃东北烤肉）的 AI 运营助手。启动时是纯前端 mock 原型，10 天内已经把下面第 1/2/3/7 项提前做完了：

- 7 个页面全部实现：今天 / 上新助手 / 门店体检 / 套餐设计 / 内容生成 / 评价管理 / AI 助手
- **真实后端 + 数据库（对应第 1 项，已完成）**：`api/*.ts` 是 Vercel Serverless Functions，`db/schema.sql` 定义的 Postgres 承载门店、评价、任务、生成记录等数据；前端启动时从 `/api/*` 拉取，不再依赖 `localStorage`（`localStorage` 现在只存当前门店和未提交的上新草稿）
- **真实文案/配图生成（对应第 2、3 项，已完成）**：上新助手和内容生成的文案调用 OpenAI 真实生成，配图调用 OpenAI GPT Image 2 或 Gemini（并会先从参考图里提取"品牌风格"再生成风格一致的配图）；调用失败时会自动降级回原来的模板拼接逻辑，不会白屏
- **仍是模板/关键词实现，尚未接真实 AI**：评价的情感分析与摘要（`analyzeReviews`）、AI 助手的问答（`generateAssistantReply`）——这两个现在还是纯本地逻辑，没有调用任何 API；门店体检和"今天"页的内容也不是 AI 生成，是 Postgres 里的种子数据
- **访问权限（对应第 7 项，部分完成）**：全站共享访问码鉴权（`AccessGate` + 每个 API 路由的 `withHandler`），并对配图这类调用真实付费 API 的接口做了按 IP 限流；Vercel 层面的 Deployment Protection 尚未确认
- 已部署：GitHub 仓库 `leoriohope/roast-ai-manager`，GitHub Actions 做 CI（typecheck + build），Vercel 托管，`dev`/`uat`/`main` 三分支对应三个环境，各自独立配置 `DATABASE_URL`/`OPENAI_API_KEY`/`GEMINI_API_KEY`/`ACCESS_CODE`

**这一步的定位**：已经不只是"演示用原型"——试点客户看到的是接了真实后端和真实 AI 生成的版本，但登录角色区分、真实评价数据接入、多租户等仍未开始（见下方第 4/5/6/8 项）。

## 后续子项目拆分

按建议顺序排列，每个都可以独立启动、验收；带 → 的是明显的依赖关系。

### 1. 后端服务 + 数据库 ✅ 已完成
把 `localStorage` 换成真正的持久化。搭一个最小后端（推荐 Vercel Serverless Functions，不用额外起服务器）+ 一个数据库（Supabase/Postgres 都行），先把现有的数据结构（门店、任务、生成记录、评价）原样搬过去，先不接真实 AI。
**验收标准**：多个设备/浏览器打开看到同一份数据；刷新、清缓存不丢数据。
**为什么排第一**：后面接真实 AI API 时，API key 不能放在前端，必须经过后端代理——先把后端搭好，第 2、3 步才有地方落地。
**现状**：`api/*.ts`（Vercel Serverless Functions）+ Postgres（`db/schema.sql`，`postgres` 库自动做 snake_case↔camelCase 转换）已上线，`localStorage` 现在只保留当前门店和未提交的上新草稿两项。

### 2. 真实文案生成接入 → 依赖 1 ✅ 已完成
把 `generateLaunchPlan` / `generatePackagePlan` / `generateContentPlan` 里的模板拼接逻辑，换成真实调用大模型。
**验收标准**：同样的输入，每次生成的文案内容真的不一样、有实际"理解"输入信息的质量提升，而不是关键词替换。
**改动范围**：主要在 `src/ai/*.ts` 内部实现，之前设计的"seam"应该能让这一步改动面很小。
**现状**：上新助手和内容生成的文案已接入 OpenAI（`api/_lib/contentCopy.ts`），并针对抖音/小红书/朋友圈/微信群做了不同的语气指导；调用失败会自动降级回模板拼接，不影响可用性。**尚未接入**：`analyzeReviews`（评价情感分析/摘要）和 `generateAssistantReply`（AI 助手问答）仍是纯本地逻辑，没有调用任何模型。

### 3. 真实图片生成接入 → 依赖 1 ✅ 已完成
把 `generateContentImage` 的 SVG 占位图换成真实图像生成 API（比如 DALL·E、Stable Diffusion 等）。
**验收标准**：生成的图能实际用于宣传（不只是渐变色+emoji）。
**备注**：这个优先级可以比第 2 步低——先把文案做真实，图片可以晚一点，看试点客户反馈里"配图"到底有没有人在意。
**现状**：已接入 OpenAI GPT Image 2 和 Gemini（`gemini-3.1-flash-image`）两个供应商，且新增了"品牌风格提取"——先用 Gemini 分析参考图（色调/打光/构图/摆盘/氛围），再把提炼出的风格注入生成配图的 prompt，保证生成图和品牌调性一致。生成的图存 Vercel Blob，数据库里只存 URL。

### 4. 用户登录与角色区分 → 依赖 1
最初的产品设想里有三种角色（老板/店长/员工），目前原型没有登录，人人看到的都一样。加最基础的登录 + 角色权限：老板看营收和门店对比，店长看今日任务和内容生成，员工可能只需要看话术。
**验收标准**：不同角色登录后默认页面/可见内容不同。

### 5. 真实评价数据接入 → 依赖 1
现在的评价是写死的 mock 数据。这一步是从美团/点评/抖音等平台把真实评价拉进来——大概率没有官方开放 API，可能得先做成"人工导入 CSV/Excel"这种半自动方案，而不是自动抓取（抓取涉及平台合规问题，需要单独评估）。
**验收标准**：评价分析页面上看到的是这个客户门店的真实评价，而不是示例数据。

### 6. 多门店/多品牌支持（多租户）→ 依赖 1
现在 4 个门店是硬编码在 `mock/stores.ts` 里，且整个产品是为"一个 BBQ 品牌"定制的措辞（"帮我做上新"这类文案跟烤肉强相关）。如果要卖给其他类型餐厅或者多个品牌，需要把品牌/门店变成可配置的租户数据，而不是写死。
**验收标准**：新增一个不同品类的门店（比如火锅店），不用改代码就能用。
**备注**：这一步如果只服务这一个试点客户，可以往后放，甚至可能永远不做。

### 7. 访问权限与安全加固 🟡 部分完成
现在部署的几个 URL（prod/dev/uat）是公网可访问、无密码的。等接了真实客户数据（尤其是第 5 步之后），需要：Vercel Deployment Protection（加密码或登录门槛）、后端 API key 通过环境变量管理、必要的速率限制。
**验收标准**：非授权用户打不开 dev/uat，prod 至少有基础访问控制。
**现状**：应用层已经有共享访问码鉴权（`AccessGate` 拦截前端、`withHandler` 拦截每个 API 路由）和按 IP 的调用限流（`api/_lib/rateLimit.ts`，目前只用于配图接口）。后端 API key 已通过 Vercel 环境变量管理，未提交到仓库。**尚未确认**：Vercel 平台层的 Deployment Protection 是否已对 dev/uat 开启。

### 8.（远期）多 Agent 架构重构
最初产品构想里提到未来可能拆成多个专职 Agent（门店诊断 Agent、上新 Agent、套餐设计 Agent、评价分析 Agent、风险审核 Agent、周报 Agent 等），现在的 `ai/` 目录是扁平的函数集合。等第 2 步真实 AI 接入、且发现单一 prompt 处理不了复杂度时，再考虑要不要往这个方向重构，现在不用提前设计。

---

## 建议的启动顺序

原计划路径是 **1 → 2 → 5**（后端 + 真文案 + 真评价）。1、2 已经完成，3（真配图）和 7（访问权限，部分）也提前做了——比原计划走得更快。**当前最有说服力的下一步是第 5 项（真实评价数据接入）**：现在后端、真文案、真配图都已经是真的，评价管理页面却还是示例数据，这是离"这真的是在用我们门店的数据"最近的一块空白。4、6、8 可以继续往后放，按实际反馈决定要不要做、什么时候做。

每个子项目开始前，我们可以单独过一遍这个子项目的具体技术方案（类似之前给 MVP 原型走的 plan mode 流程），确认范围和验收标准后再动手，避免一次性铺太大摊子。
